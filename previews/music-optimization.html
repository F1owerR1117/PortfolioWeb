🔴 P0 — 每次操作全量重新渲染

❌ 现状

编辑歌名、删除歌曲、添加封面、添加到歌单 —— 任何一个操作完成后都调用：

Components.renderMusicLibrary();

这会导致：

  • 重新请求 GET /api/music/songs + GET /api/music/playlists
  • 整个页面 innerHTML 被替换
  • 所有事件监听器重新绑定
  • 滚动位置丢失
  • 正在播放的动画重置

✅ 建议

改为局部 DOM 更新：

  • 编辑歌名 → 只更新 .song-name 的 textContent
  • 删除歌曲 → 只 remove 对应的 .song-row
  • 添加封面 → 只替换对应的 <img> src
  • 添加到歌单 → toast 提示即可，不需刷新数据

仅在新增歌曲上传成功后才全量刷新（因为需要后端返回的新 id）。

🔴 P0 — 上传逻辑嵌入 render 函数

❌ 现状

renderSongs() 函数内部直接写了：

  • 创建 <input type="file"> 元素
  • 监听 change 事件
  • 调用 showPrompt() 两次（歌名 + 艺术家）
  • 手动构造 XMLHttpRequest
  • 手动处理 upload.onprogress 更新按钮文字
  • 手动处理 onload / onerror / ontimeout

这个函数已经 200+ 行，上传逻辑占了 ~60 行。

🟡 P2 — 缺少播放队列可见性

❌ 现状

底部播放器只显示当前这一首歌。用户不知道：

  • 接下来播什么
  • 队列里有多少首歌
  • 当前播放模式（除了一个小小的 mode 按钮）

这是任何音乐播放器的核心体验缺失。

✅ 建议：点击封面展开队列面板

底部播放器封面区域加一个向上箭头的按钮，点击后从底部滑出半屏队列面板：

  • 显示当前队列前 5 首 + "还有 N 首"
  • 每首歌旁边有拖拽手柄（未来可排序）
  • 点击队列中的歌曲直接切换

这个改动约 60 行 JS + 40 行 CSS，体验提升巨大。

🟡 P2 — 播放器全量 innerHTML 重绘

❌ 现状

MusicPlayer._renderPlayer() 每次调用都执行：

container.innerHTML = [...]; this._bindPlayerEvents();

问题：

  • 播放进度条每秒更新一次 → _renderPlayer 虽未在此调用，但 _updateProgress 只更新宽度
  • 切歌 / 暂停 / 播放 / 切模式时 → 整个播放器 DOM 被替换
  • 进度条 <input type="range"> 重新创建 → seek 事件重新绑定

✅ 建议：局部 DOM 更新

首次渲染用 innerHTML，后续更新只改具体元素：

  • 歌名 → el.textContent = song.name
  • 封面 → img.src = song.cover_url
  • 播放/暂停按钮 → 只改 textContent（⏸ / ▶）
  • 模式按钮 → 只改 className 和 title
  • 进度条 → 本来就是 _updateProgress() 局部更新，保持不变
💡 总结：你的音乐界面 CSS 写得不错（playing 动画、hover 效果、卡片色彩），问题集中在 JS 架构——把所有逻辑塞进一个 render 函数里。把这 200 行的 renderMusicLibrary 拆成 5-6 个小函数，改成局部 DOM 更新，体验会提升一个档次。以上改动都不需要重写 CSS。
